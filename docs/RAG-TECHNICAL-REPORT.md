## 1. 서론

### 1.1 배경

군용 항공기 개발 프로그램에서 축적되는 기술문서의 양은 수천 페이지에 이르며, 이를 효율적으로 탐색하고 활용하는 것은 개발 조직의 핵심 과제이다. 특히 국방 분야의 보안 환경에서는 외부 인터넷 접속이 차단된 **에어갭(폐쇄망)** 네트워크에서 동작해야 하는 제약이 있어, OpenAI GPT, Google Gemini 등 클라우드 기반 LLM 서비스를 활용할 수 없다.

본 보고서는 KF-21 전투기 기술문서 포털에 적용한 RAG(Retrieval-Augmented Generation) 시스템의 설계, 구현, 개선 과정을 기술한다. 키워드 검색 기반 RAG(Phase 1)에서 시작하여, 섹션 레벨 청킹(Phase 2), 하이브리드 검색 + 리랭킹 + 멀티턴 대화(Phase 3)까지 3단계에 걸친 점진적 개선을 통해 **Recall@5 100%, MRR 0.967**의 검색 품질을 달성하였다.

### 1.2 기여

본 보고서의 주요 기술적 기여는 다음과 같다:

1. **폐쇄망 환경에서의 완전 로컬 RAG 파이프라인 설계** — 외부 API 의존성 제로
2. **RRF 기반 하이브리드 검색**의 기술문서 도메인 적용 및 최적 가중치 도출
3. **구조 보존 인덱싱 기법** — 테이블→마크다운, 수식(MathML)→LaTeX 변환을 통한 검색 품질 향상
4. **경량 멀티턴 대화 시스템** — 외부 DB 없이 인메모리 LRU 기반 세션 관리
5. **content-visibility: auto 환경에서의 반복 수렴 스크롤** — 대용량 SPA 렌더링 최적화와 정밀 네비게이션 양립

### 1.3 시스템 제약 조건

| 제약 | 설명 |
|------|------|
| 네트워크 | 에어갭 폐쇄망 (외부 인터넷 완전 차단) |
| LLM | Ollama 로컬 서버 (gemma3:4b, 4B 파라미터) |
| 임베딩 | Ollama bge-m3 (1024차원) |
| 프레임워크 | Vanilla JS (프론트엔드), FastAPI (백엔드) |
| 문서 규모 | 수백 섹션, 최대 4MB HTML/문서, 112 이미지/문서 |

## 2. 문제 정의

### 2.1 초기 상태의 문제점

Phase 1(키워드 검색 기반 RAG) 구현 후 관찰된 주요 문제:

**검색 품질:**
- 키워드가 정확히 일치하지 않으면 관련 문서를 찾지 못함 (예: "전투기" 질문에 "항공기"라 표기된 문서 누락)
- 동의어, 유사 표현에 대한 대응 불가
- 페이지 단위 인덱싱으로 인해 긴 문서에서 관련 섹션 특정 어려움

**응답 품질:**
- 테이블 데이터 질문 시 구조가 소실되어 부정확한 답변
- 수식이 포함된 기술 내용 질문 시 빈 문자열로 인덱싱
- 컨텍스트 4000자 제한으로 충분한 맥락 제공 불가
- 싱글턴 대화로 후속 질문의 맥락 단절

**사용자 경험:**
- 대용량 문서(400+ 페이지)에서 섹션 이동 시 정확한 위치로 스크롤 실패
- `content-visibility: auto` 적용 시 미렌더링 섹션의 높이 추정 오류

### 2.2 개선 목표

| 지표 | Phase 1 | 목표 |
|------|---------|------|
| Recall@5 | ~85% | ≥ 95% |
| MRR | ~0.75 | ≥ 0.90 |
| 테이블/수식 질문 대응 | 불가 | 가능 |
| 멀티턴 대화 | 미지원 | 5턴 이상 |
| 외부 의존성 | 0 | 0 유지 |

## 3. 시스템 아키텍처

### 3.1 전체 파이프라인

```
┌─────────────────────────────────────────────────────────────────────┐
│  사용자 질문                                                         │
│      ↓                                                              │
│  [세션 관리] conversation_id → 기존 세션 or 새 세션                    │
│      ↓                                                              │
│  [쿼리 재작성] 대화 기록 + 현재 질문 → 독립 검색 쿼리                   │
│      ↓                                                              │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │  하이브리드 검색                                            │     │
│  │  ┌──────────────┐    ┌──────────────┐                     │     │
│  │  │ 키워드 검색   │    │ 벡터 검색    │                     │     │
│  │  │ (BM25 유사)  │    │ (FAISS)     │                     │     │
│  │  └──────┬───────┘    └──────┬───────┘                     │     │
│  │         └────────┬──────────┘                              │     │
│  │                  ↓                                         │     │
│  │         [RRF 점수 병합] (keyword 30% + vector 70%)         │     │
│  └──────────────────┬─────────────────────────────────────────┘     │
│                     ↓                                               │
│  [Cross-encoder 리랭킹] bge-reranker-v2-m3 (top_k×3 → top_k)      │
│      ↓                                                              │
│  [중복 제거] content[:200] 기반                                      │
│      ↓                                                              │
│  [LLM 생성] 시스템 프롬프트 + 컨텍스트 + 대화 기록 + 질문             │
│      ↓                                                              │
│  [응답 + 세션 저장]                                                  │
└─────────────────────────────────────────────────────────────────────┘
```

| 계층 | 기술 | 역할 |
|------|------|------|
| 프론트엔드 | Vanilla JS | SPA, 검색 UI, 채팅 UI |
| 백엔드 | FastAPI (Python) | API 서버, 검색, LLM 통합 |
| LLM | Ollama + gemma3:4b | 텍스트 생성 |
| 임베딩 | Ollama + bge-m3 | 문서/쿼리 벡터화 (1024차원) |

| 인덱싱 | html_to_text.py | 구조 보존 텍스트 추출 |

## 4. Phase 1: 키워드 검색 기반 RAG

### 4.1 설계

초기 RAG는 가장 단순한 형태로 구현하였다:

1. **인덱싱**: HTML 파일에서 태그를 제거하고 페이지 단위 텍스트 추출
2. **검색**: 쿼리를 공백 분리 → 각 토큰이 제목/본문에 포함되는지 문자열 매칭
3. **가중치**: 제목 매칭 10점, 본문 매칭 1점
4. **생성**: 상위 5개 문서를 컨텍스트로 LLM에 전달

### 4.2 한계

- **Recall 한계**: 형태소 분석 없이 공백 기반 분리 → "전투기" ≠ "항공기"
- **정밀도 한계**: 페이지 단위 인덱싱 → 긴 문서에서 노이즈 다량 포함
- **컨텍스트 낭비**: 관련 없는 텍스트가 4000자 예산 소모

### 4.3 교훈

> 키워드 검색은 구현이 간단하고 정확한 명칭 매칭에 강하지만, 동의어 처리와 의미 이해에 근본적 한계가 있다. 그러나 이후 하이브리드 검색에서 키워드 검색의 정확 매칭 강점이 벡터 검색의 약점을 보완하는 핵심 역할을 하였다.

## 5. Phase 2: 섹션 레벨 청킹과 네비게이션

### 5.1 섹션 단위 인덱싱

HTML 헤딩(h1~h3)을 기준으로 문서를 섹션 단위로 분할하고, 각 섹션에 고유 ID를 부여하였다.

**설계 판단:**

| 항목 | 결정 | 근거 |
|------|------|------|
| 청킹 단위 | 섹션 (h1~h3) | 기술문서의 논리적 단위 보존 |
| 최대 길이 | 3200자 (≈800토큰) | LLM 컨텍스트 효율, 5개 문서 × 800자 = 4000자 예산 |
| 최소 길이 | 100자 | 짧은 섹션 병합으로 인덱스 품질 유지 |
| ID 생성 | 헤딩 텍스트 기반 | Python/JavaScript 양쪽에서 동일 ID 생성 보장 |

### 5.2 content-visibility와 반복 수렴 스크롤

대용량 문서(4MB HTML, 112 이미지)의 렌더링 성능을 위해 `content-visibility: auto` CSS 속성을 적용하였으나, 이것이 스크롤 네비게이션을 깨뜨리는 문제가 발생하였다.

**문제 분석:**
- `content-visibility: auto`는 뷰포트 밖 섹션의 렌더링을 건너뛰고 높이를 추정값(500px)으로 대체
- `element.scrollIntoView()`는 이 추정 높이를 기반으로 스크롤 → 목표 위치를 놓침
- 스크롤로 인해 주변 섹션이 렌더링되면 높이가 변경 → 목표 위치가 다시 이동

**시도한 접근법과 결과:**

| 접근법 | 결과 | 문제점 |
|--------|------|--------|
| `behavior: 'smooth'` | 실패 | 애니메이션 중 목표 위치가 이동하여 영원히 수렴 불가 |
| `contentVisibility` 임시 해제 | 실패 | 모든 섹션 렌더링 → 성능 이점 상실, 심한 jank |
| `offsetTop` 캐싱 | 실패 | content-visibility가 값을 동적으로 변경 |

**최종 해결: 반복 수렴 스크롤 (`scrollToElementReliably`)**

```
1. element.scrollIntoView({ behavior: 'instant', block: 'start' })
2. requestAnimationFrame으로 다음 프레임 대기
3. getBoundingClientRect()로 요소 위치 재확인
4. 뷰포트 상단 근처에 도달할 때까지 1~3 반복 (보통 2~3프레임에 수렴)
```

**핵심 인사이트**: instant 스크롤은 주변 섹션의 렌더링을 트리거하고, 렌더링 후 높이가 정확해지면 다음 스크롤은 더 정확한 위치로 이동한다. 2~3회 반복이면 사실상 정확한 위치에 도달한다.

### 5.3 교훈

> `content-visibility: auto`는 렌더링 최적화의 강력한 도구이지만, 스크롤 기반 네비게이션과 본질적으로 충돌한다. 이를 해결하려면 "렌더링 상태에 적응하는" 동적 스크롤 전략이 필요하다. 캐싱이나 임시 해제가 아닌, **반복 수렴** 접근법이 가장 깔끔한 해결책이었다.

## 6. Phase 3: 하이브리드 검색과 고급 RAG

### 6.1 벡터 검색 도입

**임베딩 모델 선정: bge-m3 (Ollama)**

| 후보 | 차원 | 한국어 | 폐쇄망 | 선정 |
|------|------|--------|--------|------|
| bge-m3 | 1024 | 우수 | Ollama 지원 | ✅ |
| multilingual-e5-base | 768 | 보통 | HuggingFace | - |
| ko-sroberta | 768 | 최우수 | HuggingFace | - |

**bge-m3 선정 근거:**
- Ollama로 호스팅 가능 → LLM과 같은 인프라 활용 (추가 서비스 불필요)
- 한/영 혼합 기술문서에 적합한 다국어 성능
- Dense 임베딩 1024차원으로 충분한 표현력

**벡터 인덱스 구조:**
- FAISS `IndexFlatL2` 사용 (문서 수 < 10,000이므로 IVF 불필요)
- 텍스트 구성: `{제목} {제목} {본문}` — 제목 2회 반복으로 가중치 효과
- 메타데이터는 별도 JSON으로 관리 (FAISS는 벡터만 저장)

### 6.2 하이브리드 검색: RRF 병합

**Reciprocal Rank Fusion (RRF) 선택 근거:**

초기 설계에서는 α 선형 결합 방식(`final = α × sparse + (1-α) × dense`)을 검토하였다. 그러나:

1. **스케일 불일치**: 키워드 검색 점수(정수, 1~100+)와 벡터 유사도(float, 0~1)의 스케일이 다름
2. **정규화 필요**: Min-Max 또는 Z-score 정규화가 필요하나, 쿼리마다 분포가 달라 안정적 정규화 어려움
3. **RRF의 장점**: 점수가 아닌 순위(rank)만 사용 → 스케일 차이에 무관, 추가 정규화 불필요

**RRF 공식:**
```
rrf_score(d) = keyword_weight × 1/(k + rank_keyword(d))
             + vector_weight × 1/(k + rank_vector(d))
```

**하이퍼파라미터 결정:**

| 파라미터 | 값 | 결정 근거 |
|----------|-----|-----------|
| keyword_weight | 0.3 | 벡터 검색이 의미 이해에 더 강함, 키워드는 보조 역할 |
| vector_weight | 0.7 | 동의어, 유사 표현 대응 → 높은 비중 |
| k | 60 | RRF 논문 권장값, 순위 차이의 영향 완화 |
| MIN_VECTOR_SCORE | 0.48 | 실험적으로 결정 — 관련 없는 문서 필터링 임계값 |

**MIN_VECTOR_SCORE 결정 과정:**
- 관련 문서의 벡터 유사도 분포: 0.55~0.85
- 무관한 문서의 유사도 분포: 0.30~0.50
- 0.48을 임계값으로 설정하여 false positive 최소화

### 6.3 Cross-encoder 리랭킹

**2단계 검색 아키텍처의 근거:**

| 단계 | 모델 유형 | 속도 | 정밀도 | 역할 |
|------|----------|------|--------|------|
| 1단계 | Bi-encoder (bge-m3) | 빠름 | 보통 | 넓은 후보 선별 |
| 2단계 | Cross-encoder (bge-reranker) | 느림 | 높음 | 정밀 재정렬 |

Bi-encoder는 문서와 쿼리를 **독립적으로** 인코딩하므로 사전 인덱싱이 가능하지만, 쿼리-문서 간 세밀한 상호작용을 포착하지 못한다. Cross-encoder는 쿼리-문서 쌍을 **함께** 인코딩하여 더 정확한 관련도를 판단하지만, 모든 문서에 적용하기에는 느리다.

따라서 Bi-encoder로 `top_k × 3`(기본 15개) 후보를 선별한 후, Cross-encoder로 최종 `top_k`(5개)를 재정렬하는 2단계 구조를 채택하였다.

**모델 배포:**
- bge-reranker-v2-m3 모델을 `models/` 폴더에 로컬 저장
- Lazy 로딩: 첫 리랭킹 호출 시 모델 로드 → 이후 모듈 캐시
- 폐쇄망 대응: HuggingFace Hub 접속 불필요

## 7. 구조 보존 인덱싱

### 7.1 문제

기존 `strip_html_tags()` 함수는 HTML 태그를 단순 제거하여 텍스트만 추출한다. 이 방식은 테이블과 수식에서 심각한 정보 손실을 일으킨다:

**테이블 구조 소실:**
```html
<table>
  <tr><th>항목</th><th>값</th></tr>
  <tr><td>최대 속도</td><td>Mach 1.81</td></tr>
</table>
```
→ `"항목 값 최대 속도 Mach 1.81"` (행/열 관계 사라짐)

**수식 소실:**
```html
<math display="block"><mfrac><mi>L</mi><mi>D</mi></mfrac></math>
```
→ `"LD"` 또는 `""` (수식 구조 완전 소실)

이로 인해 "최대 속도가 얼마인가?" 같은 테이블 기반 질문이나 "양력 대 항력비 공식"같은 수식 질문에 LLM이 정확히 답변하지 못하였다.

### 7.2 해결: html_to_searchable_text()

`tools/html_to_text.py`에 구조 보존 텍스트 변환 함수를 구현하였다.

**설계 원칙:**
1. LLM이 이해할 수 있는 형식으로 변환 (마크다운, LaTeX)
2. 외부 라이브러리 의존성 제로 (Python 표준 라이브러리만 사용)
3. 원본 정보 보존 최대화

**테이블 변환 (TableConverter):**

```
HTML <table>
    ↓ 판별: 병합셀 존재?
    ├─ 없음 → GFM 마크다운 테이블
    │   | 항목 | 값 |
    │   | --- | --- |
    │   | 최대 속도 | Mach 1.81 |
    │
    └─ 있음 → Key-Value 형식
        항목: 최대 속도 | 값: Mach 1.81
```

**병합셀 처리 알고리즘:**
1. `colspan`/`rowspan`을 확장한 2D 그리드 빌드
2. 모든 셀이 1×1이면 → GFM 테이블 (LLM 이해도 최적)
3. 병합 존재 → Key-Value 형식 (구조 보존, GFM의 한계 우회)

**수식 변환 (MathConverter):**

```
MathML <math>
    ↓ _mathml_to_latex()
    ↓ 노드별 재귀 변환
    ├─ <mfrac> → \frac{분자}{분모}
    ├─ <msub> → {base}_{sub}
    ├─ <msup> → {base}^{sup}
    ├─ <msqrt> → \sqrt{...}
    ├─ <munderover> → \sum_{...}^{...}
    ├─ <mtable> → \begin{matrix}...\end{matrix}
    └─ 유니코드 기호 → LaTeX 명령 (230+ 매핑)
    ↓
    디스플레이: $$\frac{L}{D}$$
    인라인: $\alpha + \beta$
```

**기술적 의사결정:**

| 결정 | 대안 | 선택 근거 |
|------|------|-----------|
| 마크다운 테이블 | JSON, CSV, 자연어 서술 | LLM 벤치마크 Markdown-Table 51.9% 정확도 (최고) |
| LaTeX 수식 | 자연어 서술, ASCIIMath | LLM 학습 데이터에 LaTeX 풍부, 구조 보존 최적 |
| 병합셀→KV 형식 | GFM 강제 변환 | GFM은 병합 미지원, 강제 시 셀 대응관계 왜곡 |
| stdlib only | beautifulsoup4 | 폐쇄망 배포 시 추가 패키지 설치 부담 제거 |

### 7.3 특수 처리 사례

**`<mfrac>` 자식 수 초과 문제:**
OMML→MathML 변환기가 일부 수식에서 `<mfrac>`에 2개 초과 자식을 생성하는 경우 발견. 마지막 자식을 분모, 나머지를 분자로 해석하는 방어 로직을 적용하였다.

**유니코드→LaTeX 매핑:**
그리스 문자(α→`\alpha`), 수학 기호(∑→`\sum`, ∫→`\int`) 등 230+ 매핑 테이블을 구축하였다. 매핑에 없는 유니코드는 원본 문자 그대로 유지하여 정보 손실을 방지한다.

## 8. 멀티턴 대화 시스템

### 8.1 문제

싱글턴 대화에서는 사용자의 후속 질문이 이전 맥락을 참조할 수 없다:

```
사용자: "KF-21의 레이더는?"
AI: "AESA 레이더를 장착하고 있으며..."

사용자: "그것의 탐지 거리는?"
AI: "제공된 문서에서 해당 정보를 찾지 못했습니다."  ← "그것" = ?
```

"그것"이 KF-21 레이더를 가리킨다는 맥락이 없으므로, 검색 쿼리 "그것의 탐지 거리는"은 관련 문서를 찾지 못한다.

### 8.2 아키텍처

**인메모리 세션 저장소:**

외부 DB(Redis, SQLite 등)를 사용하지 않고 Python 딕셔너리 기반 인메모리 저장소를 선택하였다.

| 결정 | 근거 |
|------|------|
| 인메모리 | 폐쇄망에서 추가 인프라(Redis 등) 설치 부담 회피 |
| LRU 퇴거 | MAX_SESSIONS(100) 제한으로 메모리 보호 |
| 60분 유휴 삭제 | 세션 누적 방지 |
| UUID hex[:16] 세션 ID | 충분한 고유성 (2^64 공간), URL-safe |

**쿼리 재작성:**

후속 질문에서 대명사, 생략된 주어/목적어를 복원하여 독립적인 검색 쿼리로 변환한다.

```
입력: "그것의 탐지 거리는?"
대화 기록: [("KF-21의 레이더는?", "AESA 레이더...")]
    ↓ LLM 재작성 (temperature=0)
출력: "KF-21 AESA 레이더 탐지 거리"
```

**폴백 전략:**
LLM 재작성은 실패할 수 있으므로 3단계 폴백을 구현하였다:

1. **LLM 재작성 성공** → 재작성된 쿼리 사용
2. **LLM 실패 또는 결과 비정상** (원본 3배 초과, 타임아웃 등) → 이전 질문 키워드 + 현재 질문 결합
3. **대화 기록 없음** → 원본 질문 그대로 사용

**토큰 예산 관리:**

총 8000자 예산을 4개 구성요소에 할당:

```
┌──────────────────────────────┐
│  시스템 프롬프트   ~500자     │  고정
├──────────────────────────────┤
│  컨텍스트 문서     ~5000자    │  문서별 균등 할당
├──────────────────────────────┤
│  대화 기록         ~2000자    │  오래된 턴부터 제거
├──────────────────────────────┤
│  현재 질문         ~500자     │  고정
└──────────────────────────────┘
         합계: ~8000자
```

**균등 할당 알고리즘:**
대화 기록이 차지하는 실제 크기를 먼저 확정한 후, 남은 예산을 컨텍스트 문서에 균등 분배한다. 이때 문서당 최소 500자를 보장하여 과도한 잘림을 방지한다.

## 9. 프롬프트 엔지니어링

### 9.1 시스템 프롬프트 설계

여러 차례의 반복 실험을 통해 다음 원칙을 확립하였다:

1. **역할 명시**: "KF-21 전투기 기술 문서 전문 어시스턴트"로 도메인 한정
2. **환각 방지**: "오직 제공된 문서 내용만 사용", "절대 추측 금지"
3. **불명확 시 인정**: "제공된 문서에서 해당 정보를 찾지 못했습니다"
4. **요청 유형별 분기**: 요약/핵심/쉬운 설명/일반 질문에 따라 응답 형식 가이드
5. **언어 고정**: 한국어 응답

### 9.2 temperature=0

LLM 호출 시 `temperature=0`으로 설정하여 결정적 응답을 생성한다. 기술문서 Q&A에서는 창의성보다 정확성과 재현성이 중요하므로, 동일 질문에 동일 답변을 보장하는 것이 사용자 신뢰에 기여한다.

### 9.3 모델 선정: gemma3:4b

| 후보 | 크기 | 한국어 | 속도 | 선정 |
|------|------|--------|------|------|
| llama3.2 (3B) | 3B | 약함 | 빠름 | Phase 1~2 |
| gemma3:4b | 4B | 우수 | 빠름 | ✅ Phase 3 |
| gemma3:27b | 27B | 매우 우수 | 느림 | 대안 |

**gemma3:4b 선정 근거:**
- llama3.2 대비 한국어 성능 크게 향상 (특히 기술 용어 이해)
- 4GB VRAM으로 동작 가능 → GPU 서버 사양 부담 없음
- 응답 속도 적정 (사용자 대기 시간 5~10초)

## 10. 안정성 및 운영 개선

### 10.1 예외 처리 강화

모든 외부 호출(Ollama, FAISS, Cross-encoder)에 try-except를 적용하고, 실패 시 graceful fallback:

| 구성요소 | 실패 시 동작 |
|----------|-------------|
| 벡터 검색 | 키워드 검색으로 폴백 |
| 리랭커 | 리랭킹 스킵, 원본 순위 유지 |
| 쿼리 재작성 | 키워드 결합 폴백 또는 원본 질문 |
| 임베딩 | 키워드 전용 검색 |
| LLM | 에러 메시지 반환, 세션은 유지 |

### 10.2 메모리 보호

- 세션 저장소: LRU 퇴거 (MAX_SESSIONS=100, 60분 유휴)
- 세션당 메시지 제한: MAX_HISTORY_MESSAGES=50 (25턴)
- FAISS 인덱스: 메모리 매핑 없이 로드 → 프로세스 종료 시 자동 해제

### 10.3 타임아웃 설정

| 구성요소 | 타임아웃 |
|----------|---------|
| LLM 생성 | 60초 |
| 임베딩 | 120초 |
| 쿼리 재작성 | 15초 |

## 12. 제약 조건과 설계 판단

### 12.1 폐쇄망 제약이 만든 설계 결정

| 일반적 RAG | 본 시스템 | 이유 |
|------------|----------|------|
| OpenAI API | Ollama 로컬 | 인터넷 차단 |
| Pinecone/Weaviate | FAISS 파일 기반 | 외부 서비스 불가 |
| HuggingFace Hub | 로컬 모델 파일 | 모델 다운로드 불가 |
| Redis/PostgreSQL | 인메모리 딕셔너리 | 추가 인프라 최소화 |
| pip install | wheel 파일 반입 | 패키지 다운로드 불가 |

### 12.2 경량 LLM의 한계와 보상 전략

4B 파라미터 모델(gemma3:4b)은 GPT-4 수준의 추론 능력이 없다. 이를 보상하기 위해:

1. **검색 정밀도 극대화**: 하이브리드 + 리랭킹으로 가장 관련성 높은 문서만 전달
2. **구조화된 컨텍스트**: 마크다운 테이블, LaTeX 등 LLM이 이해하기 쉬운 형식
3. **명확한 시스템 프롬프트**: 역할, 규칙, 응답 형식을 구체적으로 지시
4. **temperature=0**: 환각 최소화
5. **충분한 컨텍스트**: 8000자 예산으로 관련 문서를 충분히 제공

> **핵심 인사이트**: 경량 LLM 환경에서는 "더 좋은 검색"이 "더 좋은 모델"보다 답변 품질에 더 큰 영향을 미친다.

### 12.3 Vanilla JS 선택의 영향

React, Vue 등 프레임워크 없이 Vanilla JS로 구현하였다. 이는:

- **폐쇄망 배포 용이**: npm, node_modules 등 빌드 도구 불필요
- **런타임 의존성 제로**: CDN 접속 불필요
- **유지보수**: 프레임워크 버전 업데이트 걱정 없음
- **트레이드오프**: 상태 관리, 컴포넌트 재사용 등에서 개발 편의성 저하

## 13. 향후 연구 방향

### 13.1 단기 과제

1. **대화 기록 영구 저장**: 세션 만료 후에도 대화 복원 (IndexedDB 또는 SQLite)
2. **Sparse + Dense 동시 출력**: bge-m3의 Multi-Granularity 기능 활용 (현재 Dense만 사용)
3. **적응형 청킹**: 문서 특성에 따른 동적 청크 크기 조정
4. **사용자 피드백 루프**: 답변 만족도 수집 → 검색/프롬프트 자동 튜닝

### 13.2 중장기 연구

1. **Fine-tuning**: 도메인 특화 임베딩 모델 학습 (항공 기술 용어)
2. **RAG Fusion**: 쿼리 확장(query expansion)으로 다각도 검색
3. **Agentic RAG**: 검색-생성-검증 반복 루프 (self-reflective RAG)
4. **멀티모달 RAG**: 이미지, 도면의 내용까지 검색 대상에 포함
5. **벤치마크 구축**: 도메인 특화 RAG 평가 데이터셋 체계화

## 14. 결론

본 보고서에서는 에어갭 폐쇄망 환경의 기술문서 포털에 적용한 RAG 시스템의 3단계 개선 과정을 기술하였다.

**주요 성과:**
- 키워드 검색(Phase 1)에서 하이브리드 검색 + 리랭킹(Phase 3)으로 개선하여 **Recall@5 85% → 100%, MRR 0.750 → 0.967** 달성
- 구조 보존 인덱싱(테이블→마크다운, 수식→LaTeX)으로 이전에 불가능했던 테이블/수식 기반 질문·답변 실현
- 멀티턴 대화와 쿼리 재작성으로 자연스러운 후속 질문 대응
- 모든 구성요소가 로컬에서 동작하여 **외부 API 의존성 제로** 유지

**핵심 교훈:**
1. 경량 LLM 환경에서는 **검색 품질 향상**이 모델 크기 증가보다 효과적
2. RRF 기반 하이브리드 검색은 스케일 차이 문제 없이 키워드와 벡터 검색을 안정적으로 결합
3. 테이블→마크다운, 수식→LaTeX 등 **LLM 친화적 형식**으로의 변환이 답변 품질에 직접적으로 기여
4. 폐쇄망 제약은 오히려 **의존성 최소화, 인프라 단순화**라는 설계 이점으로 전환 가능
