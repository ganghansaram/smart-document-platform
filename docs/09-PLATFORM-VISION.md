## 1. 프로젝트 발전 경과

본 프로젝트는 "KF-21 개발 백서 열람용 웹북" 시범 시스템으로 시작했으나, 개발이 진행되면서 단순 웹북을 넘어 플랫폼으로의 확장 가능성이 확인되었다.

### 발전 단계

```
Phase 1 (완료)  : 정적 웹북           → "읽는 시스템"
Phase 2 (완료)  : RAG + 편집기        → "활용하는 시스템"
Phase 3 (완료)  : 관리 기능 + 통계     → "운영하는 시스템"
Phase 4 (검토중) : 비교/번역/다중 시스템 → "기술 문서 활용 플랫폼"
```

### 현재 시스템의 확장 가능 영역

| 영역 | 현재 상태 | 확장 방향 |
|------|----------|----------|
| 문서 열람 | HTML 기반 구조적 탐색, 섹션 네비게이션 | 다중 문서 동시 열람, 문서 비교 |
| 문서 생산 | Word/PDF → HTML 자동 변환, 웹 에디터 | 실시간 협업 편집, 버전 관리 |
| 검색 | 키워드 + 벡터 하이브리드 검색, 리랭킹 | 필터 검색(태그/날짜/작성자), 유사 문서 추천 |
| AI 활용 | RAG 챗봇 (멀티턴 대화, 쿼리 재작성) | 논문 번역, 요약 생성, 문서 자동 분류 |
| 관리 | 메뉴 관리, 사용자 관리, 접속 통계 | 조직별 권한, 문서 워크플로우, 감사 로그 |

### 시스템 정의

**"기술 문서 활용 플랫폼 (Technical Document Platform)"**

| 후보 명칭 | 적합성 | 이유 |
|----------|--------|------|
| 단순 웹북 | 부족 | RAG, 편집기, 업로드, 통계까지 이미 초과 |
| RAG 시스템 | 부분적 | RAG는 기능 중 하나일 뿐 |
| 문서 포털 | 수동적 | 열람만 암시, 편집/변환/검색 미포함 |
| **기술 문서 활용 플랫폼** | **적합** | 생산(변환) → 관리(메뉴) → 열람 → 검색 → AI 활용을 포괄 |

## 2. 플랫폼 구성 — 3개 시스템

```
┌──────────────────────────────────────────────────────────┐
│                    진입점 (Launcher)                       │
│                                                          │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│   │  📖 문서 탐색   │  │  🔍 문서 비교   │  │  🌐 논문 리딩   │  │
│   │  Explorer    │  │  Compare     │  │  Reader      │  │
│   │              │  │              │  │              │  │
│   │  기술문서     │  │  diff/규칙검토│  │  원문+번역    │  │
│   │  열람/검색/AI │  │  side-by-side│  │  side-by-side│  │
│   └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                          │
│   각 시스템은 독립된 레이아웃 + 공유 백엔드                  │
└──────────────────────────────────────────────────────────┘
```

### 시스템별 성격

| | 문서 탐색 (Explorer) | 문서 비교 (Compare) | 논문 리딩 (Reader) |
|--|------|------|------|
| **목적** | 기술문서 열람/검색/AI 질의 | 문서 간 차이 분석, 규칙 준수 검토 | 영문 논문/규격서 한국어 독해 |
| **입력** | menu.json 기반 트리 | 두 문서 선택 또는 업로드 | PDF 업로드 (또는 URL) |
| **레이아웃** | 3단 (트리 + 콘텐츠 + TOC) | 2단 좌우 분할 (원본 vs 비교본) | 2단 좌우 분할 (원문 vs 번역) |
| **핵심 기술** | RAG, 벡터 검색 | diff 알고리즘, LLM 의미 비교 | PDF 파싱, LLM 번역 |
| **현재 상태** | **완성** | 신규 | 신규 |

## 3. 진입점 설계

### 권장안: 로그인 후 런처 화면

```
로그인 → 런처 (3개 카드) → 각 시스템

┌──────────────────────────────────────────────┐
│                                              │
│   📖 문서 탐색         🔍 문서 비교            │
│   기술문서를            두 문서의 차이를        │
│   탐색합니다            비교/분석합니다         │
│                                              │
│              🌐 논문 리딩                      │
│              영문 논문을                       │
│              한국어로 읽습니다                  │
│                                              │
└──────────────────────────────────────────────┘
```

- 로그인 성공 → `launcher.html`로 이동 (기존 `index.html` 대신)
- 각 카드 클릭 → 해당 시스템의 페이지로 진입
- 각 시스템 헤더에 시스템 간 전환 네비게이션 배치

### 대안

| 안 | 방식 | 장단점 |
|----|------|--------|
| **A (권장)** | 런처 페이지 | 깔끔한 분리, 각 시스템 독립 레이아웃 유지 |
| B | 웹북 헤더에 탭 추가 | 단순하지만 레이아웃 전환 복잡도 높음 |
| C | 로그인 페이지가 런처 겸용 | 로그인 후 자동 진입, 헤더에서 전환 |

## 4. 시스템 1: 문서 탐색 (Explorer)

현재 완성된 웹북 시스템. 변경 없이 그대로 유지.

### 주요 기능 (구현 완료)

- HTML 기반 구조적 문서 탐색 (트리 메뉴 + 섹션 네비게이션)
- Word/PDF → HTML 자동 변환 (수식, 표, 이미지 포함)
- 키워드 + 벡터 하이브리드 검색 (FAISS + bge-m3, RRF 병합)
- RAG 챗봇 (멀티턴 대화, 쿼리 재작성, Cross-encoder 리랭킹)
- 웹 기반 문서 편집기 (Monaco Editor)
- 관리자 설정 GUI (메뉴 관리, 사용자 관리, 접속 통계)
- 3단계 RBAC (viewer / editor / admin)

### 레이아웃

```
┌─ 헤더 ──────────────────────────────────────┐
├──────┬──────────────────────┬───────────────┤
│ 트리  │     콘텐츠 영역       │   목차 (TOC)  │
│ 메뉴  │                     │               │
│      │                     │               │
│      │                     │               │
├──────┴──────────────────────┴───────────────┤
│ 검색 / AI 챗봇 (오버레이)                     │
└─────────────────────────────────────────────┘
```

## 5. 시스템 2: 문서 비교 (Compare)

### 용도

기술문서 간 차이 분석 및 작성 규칙 준수 검토.

### 비교 모드

| 모드 | 설명 | 기술 |
|------|------|------|
| **텍스트 diff** | 두 버전 간 텍스트 변경점 표시 (추가/삭제/수정) | diff 알고리즘 (문장 단위) |
| **의미 비교** | 같은 주제를 다루지만 내용이 다른 부분 찾기 | LLM 기반 의미 유사도 비교 |
| **규칙 검토** | 문서 작성 규칙 준수 여부 체크 (번호 체계, 용어 통일 등) | 규칙 엔진 + LLM 보조 |

### 화면 레이아웃

```
┌─ 문서 비교 ─────────────────────────────────────────────┐
│  [모드: diff ▾]  [문서A 선택 ▾]  [문서B 선택 ▾]  [비교]   │
├─────────────────────┬───────────────────────────────────┤
│  문서 A (원본)        │  문서 B (비교본)                    │
│                     │                                   │
│  1.1 개요            │  1.1 개요                          │
│  본 시스템은 항공기의  │  본 시스템은 ██항공기██의            │
│  비행 제어를 담당...  │  ██비행제어장치를██ 담당...          │
│                     │                                   │
│  (동기 스크롤)        │  (동기 스크롤)                      │
├─────────────────────┴───────────────────────────────────┤
│  비교 결과 요약: 변경 12건, 추가 3건, 삭제 1건              │
│  [섹션 1.1] 용어 변경: "비행 제어" → "비행제어장치"          │
│  [섹션 2.3] 내용 추가: 신규 요구사항 3건 ...                │
└─────────────────────────────────────────────────────────┘
```

### 핵심 UX

- **좌우 동기 스크롤** — 섹션(heading) 기준 앵커 매칭
- **하단 요약 패널** — 차이점 목록, 클릭 시 해당 위치로 스크롤 이동
- **문서 소스** — 웹북 내 등록 문서 선택 또는 외부 파일 업로드
- **변경점 하이라이트** — 추가(초록), 삭제(빨강), 수정(노랑) 색상 구분

### 백엔드 API

```
POST /api/compare/diff     — 두 문서의 텍스트 diff 결과 반환
POST /api/compare/semantic  — LLM 기반 의미 비교 결과 반환
POST /api/compare/rules     — 규칙 준수 검토 결과 반환
```

## 6. 시스템 3: 논문 리딩 (Reader)

### 용도

영문 논문/규격서를 한국어로 독해하기 위한 번역 리딩 시스템.

### 참고 프로젝트

[hi-space/scholar-translator](https://github.com/hi-space/scholar-translator)
- PDF 업로드 → 레이아웃 감지 (DocLayout-YOLO) → 블록 단위 번역
- mono(번역만) / dual(원문+번역 병렬) 두 가지 출력
- AWS Bedrock Claude 기반 번역

### 차별점 (scholar-translator 대비)

| scholar-translator | 우리 시스템 |
|---|---|
| Gradio UI (범용) | 웹북 플랫폼에 통합된 전용 UI |
| PDF → PDF 출력 | PDF → **웹 뷰** (HTML 기반, 인터랙티브) |
| AWS Bedrock (클라우드) | **Ollama** (폐쇄망 호환) |
| 일괄 번역 후 다운로드 | **실시간 스트리밍** (단락별 점진 번역) |
| 정적 결과물 | 동기 스크롤 + 클릭 번역 인터랙션 |

### 화면 레이아웃

```
┌─ 논문 리딩 ─────────────────────────────────────────────┐
│  [PDF 업로드]  [번역 모델 ▾]  [전체 번역 / 선택 번역]      │
├─────────────────────┬───────────────────────────────────┤
│  원문 (English)      │  번역 (한국어)                      │
│                     │                                   │
│  1. Introduction    │  1. 서론                           │
│  The flight control │  비행 제어 시스템은 항공기의           │
│  system is...       │  ...                              │
│                     │                                   │
│  [Figure 1]  ────── │ ────── [Figure 1] (공유)            │
│                     │                                   │
│  (동기 스크롤)        │  (동기 스크롤)                      │
├─────────────────────┴───────────────────────────────────┤
│  용어 사전: flight control = 비행 제어 (수정 가능)          │
└─────────────────────────────────────────────────────────┘
```

### 핵심 UX

- **단락 기준 동기 스크롤** — 원문 단락과 번역 단락이 나란히 정렬
- **클릭 번역** — 전체 번역 대신 특정 단락 클릭 시 해당 부분만 번역 (빠른 탐색)
- **용어 사전** — 도메인 용어 매핑 테이블, 번역 일관성 유지
- **그림/표 공유** — 이미지/테이블은 번역 불필요, 양쪽에 동일하게 표시
- **번역 캐싱** — 한 번 번역한 단락은 재요청 없이 즉시 표시

### 백엔드 API

```
POST /api/pdf/parse       — PDF 업로드 → 블록(단락/표/이미지) 단위 추출
POST /api/translate       — 텍스트 블록 번역 (스트리밍 응답)
GET  /api/translate/cache — 캐싱된 번역 결과 조회
POST /api/glossary        — 용어 사전 관리 (추가/수정/삭제)
```

## 8. 구조적 리스크 및 개선 사항

### 현재 구조의 강점

1. **모듈화된 백엔드** — FastAPI 라우터 기반, 신규 모듈 추가가 파일 하나 추가 수준
2. **프론트-백 분리** — UI 변경과 백엔드 로직 변경이 독립적
3. **폐쇄망 호환** — 외부 CDN/SaaS 의존 없이 완전 자립 운영
4. **런타임 설정** — 재배포 없이 운영 환경 조정 가능
5. **AI 인프라** — Ollama 기반 파이프라인이 번역/비교로 확장 용이

### 선제적 개선이 필요한 부분

| 리스크 | 현재 상태 | 영향 | 개선 방향 |
|--------|----------|------|----------|
| 문서-메뉴 경로 종속 | URL = 파일 경로 = 메뉴 위치 | 메뉴 변경 시 문서 링크 파손 | 문서 ID 기반 저장 + 메타데이터 분리 |
| 단일 인스턴스 | 하나의 문서 컬렉션만 운영 | 타 조직 도입 시 별도 설치 필요 | 프로젝트/워크스페이스 개념 도입 |
| 파일 기반 저장 | menu.json, search-index.json | 동시 쓰기 충돌 가능성 | 핵심 메타데이터 DB화 (SQLite 확장) |

### 확장 시 우선 개선 순서

1. **문서 저장 경로의 메뉴 종속성 해소** — `doc-registry` 도입 (ID 기반 관리)
2. **콘텐츠 로딩 추상화** — `loadContent(url)` → 콘텐츠 핸들러 패턴으로 전환
3. **메타데이터 DB화** — 번역 캐시, 용어 사전, 비교 이력 등 SQLite 확장

### 재설계가 필요해지는 시점

- 동시 사용자 100명 이상 → DB 기반 세션/큐 필요
- 문서 수 1,000개 이상 → 파일 기반 → DB 기반 전환 검토
- 멀티 테넌트 (조직별 분리) → 인증/권한 체계 재설계

**현재 규모에서는 재설계 불필요. 과잉 설계가 더 큰 리스크.**

## 9. 개발 로드맵

### 단기 (Phase 4a) — 문서 비교

```
1. 런처 페이지 구현 (launcher.html)
2. 문서 비교 UI (좌우 split-pane, 동기 스크롤)
3. 텍스트 diff 백엔드 (api/compare.py)
4. LLM 의미 비교 모드
5. 규칙 검토 모드
```

문서 비교를 먼저 개발하는 이유:
- 기존 HTML 문서를 그대로 활용 가능 (PDF 파싱 불필요)
- diff 알고리즘은 성숙한 기술 (라이브러리 풍부)
- 실무 수요가 즉시 존재 (기술문서 버전 비교)

### 중기 (Phase 4b) — 논문 리딩

```
1. PDF 파싱 백엔드 (api/pdf_parse.py) — PyMuPDF 기반 블록 추출
2. 번역 API (api/translate.py) — Ollama 스트리밍 번역
3. 논문 리딩 UI (좌우 split-pane, 단락 동기 스크롤)
4. 클릭 번역 (선택 단락만 번역)
5. 용어 사전 (api/glossary.py)
6. 번역 캐싱
```

### 장기 (Phase 5) — 플랫폼 고도화

```
1. 문서 저장 구조 개선 (ID 기반 + 메타데이터 레지스트리)
2. 멀티 프로젝트 지원 (조직별 독립 문서 컬렉션)
3. 문서 워크플로우 (초안 → 검토 → 승인 → 발행)
4. 유사 문서 추천, 자동 분류
```
