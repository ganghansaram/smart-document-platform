## 1. 현황 분석

### 1.1 식별된 문제

#### 문제 1: 언어 전환 현상

쿼리 재작성(`query_rewriter.py`), Agentic RAG(`rag_agent.py`), 쿼리 분해(`query_decomposer.py`) 과정에서 LLM이 영어 쿼리를 생성하면, 영어 문서가 검색되고, 최종 응답까지 영어로 나오는 연쇄 현상이 발생한다.

**발생 경로**:
```
사용자: "관련해서 결과표값 같은게 있으면 알려줘"
    ↓ 쿼리 재작성 (LLM이 이전 답변의 영어 용어 참조)
"RAG performance benchmark results table"  ← 영어로 전환됨
    ↓ 벡터 검색 (bge-m3)
영어 문서 히트 (FY1-DoD-FLEX-4 등)  ← 영어 컨텍스트
    ↓ LLM 응답 생성
영어 답변 생성  ← 사용자는 한국어 기대
```

**원인 분석**:
- `query_rewriter.py`: REWRITE_SYSTEM_PROMPT에 "한국어로 재작성" 명시 없음 (line 13-21)
- `rag_agent.py`: PLAN_PROMPT에 언어 제약 없음 (line 18-32)
- `query_decomposer.py`: DECOMPOSE_PROMPT에 언어 제약 없음
- `llm_client.py`: "한국어로 답변합니다" 있지만 (line 39), 컨텍스트가 영어이면 gemma3:4b가 무시

#### 문제 2: 키워드 검색의 나이브한 구현

`keyword_search.py`의 현재 구현:
```python
# 공백 기반 토큰 분리
terms = [t.lower() for t in query.split() if len(t) >= 2]

# 단순 substring 매칭
if term in title_lower:
    score += 10
if term in content_lower:
    score += 1
```

**한계**:
1. **TF-IDF 미반영**: "의", "및" 같은 고빈도 조사가 "감항인증"같은 핵심어와 동일 가중치
2. **문서 길이 미고려**: 짧은 섹션(100자)과 긴 섹션(3200자)에서 단어 출현 1회를 동일 취급
3. **형태소 분석 없음**: "비행시험을" → 조사 포함 토큰, "비행시험" 검색에 불일치
4. **복합명사 미분리**: "전자전체계" → 하나의 토큰, "전자전" 검색에 미히트

#### 문제 3: 답변 검증 부재

LLM이 생성한 답변을 검증 없이 바로 반환한다. 시스템 프롬프트에 "문서에 없는 내용은 절대 추측하지 않습니다"라는 지시가 있지만, gemma3:4b가 항상 준수하지는 않는다.

기존 베이스라인 테스트 결과: 환각률 ~10% (15문항 중 ~2건에서 △ 판정)

### 1.2 현재 파이프라인 구조

```
사용자 질문
    ↓
[세션 관리] → 대화 기록 조회
    ↓
[쿼리 재작성] query_rewriter.py → Ollama (temp=0)
    ↓
[질문 라우팅] question_router.py → SIMPLE/COMPARE/REASON/CHAT
    ↓
┌─ SIMPLE ──────────────────────────────────────┐
│  [키워드 검색] ← 나이브 매칭                    │
│  [벡터 검색] ← FAISS + bge-m3                  │
│  [RRF 병합] keyword 30% + vector 70%           │
│  [리랭킹] ← bge-reranker-v2-m3                │
├─ COMPARE ─────────────────────────────────────┤
│  [쿼리 분해] → 서브쿼리 생성                    │
│  [멀티 검색] 각 서브쿼리별 하이브리드 검색        │
│  [병합 + 리랭킹]                               │
├─ REASON ──────────────────────────────────────┤
│  [Agentic RAG] → 최대 3회 반복 검색             │
│  [신뢰도 판단]                                  │
└───────────────────────────────────────────────┘
    ↓
[LLM 응답 생성] → 스트리밍
    ↓              ← 검증 없음 (Step 3에서 추가)
[응답 반환]
```

## 2. Step 1: 언어 제어 강화

### 2.1 목표

모든 LLM 호출에서 한국어 출력을 보장하여, 영어 문서가 검색되더라도 한국어로 답변하도록 한다.

### 2.2 변경 사항

#### 2.2-A. 쿼리 재작성 프롬프트 (`backend/services/query_rewriter.py`)

- **위치**: REWRITE_SYSTEM_PROMPT, line 13-21
- **변경**: 규칙 5번 뒤에 추가
  ```
  6. **반드시 한국어로 재작성** (영어 기술 용어는 예외: AESA, FAISS 등)
  ```

#### 2.2-B. Agentic RAG 계획 프롬프트 (`backend/services/rag_agent.py`)

- **위치**: PLAN_PROMPT, line 18-32
- **변경**: 규칙 끝에 추가
  ```
  - **query는 반드시 한국어로 작성** (영어 약어는 예외: AESA, BM25 등)
  ```

#### 2.2-C. 쿼리 분해 프롬프트 (`backend/services/query_decomposer.py`)

- **위치**: DECOMPOSE_PROMPT 규칙 섹션
- **변경**: 규칙 추가
  ```
  6. **서브쿼리는 반드시 한국어로 작성** (영어 약어는 예외)
  ```

#### 2.2-D. 응답 생성 시스템 프롬프트 (`backend/services/llm_client.py`)

- **위치**: DEFAULT_SYSTEM_PROMPT [언어] 섹션, line 38-39
- **현재**: `한국어로 답변합니다.`
- **변경**:
  ```
  반드시 한국어로 답변합니다. 참고 문서가 영어여도 한국어로 번역하여 답변합니다.
  영어 기술 용어는 한국어 설명 뒤 괄호 안에 병기합니다 (예: 능동전자주사배열 레이더(AESA)).
  ```

### 2.3 영향 분석

| 항목 | 영향 |
|------|------|
| 검색 결과 | 한국어 쿼리 유지 → 한국어 문서 우선 히트 |
| 응답 언어 | 영어 컨텍스트 → 한국어 답변 보장 |
| 기존 동작 | 이미 한국어 쿼리인 경우 변화 없음 |
| 성능 | 변화 없음 (프롬프트 텍스트만 변경) |

### 2.4 위험도: **매우 낮음**

프롬프트 텍스트 변경만. 롤백은 문자열 원복으로 즉시 가능.

## 3. Step 2: BM25 + 한국어 형태소 분석

### 3.1 목표

나이브 키워드 검색을 BM25 기반 확률적 검색으로 교체하고, 한국어 형태소 분석으로 토큰화 품질을 개선한다.

### 3.2 기술 선정

| 구분 | 선정 | 근거 |
|------|------|------|
| BM25 | `rank-bm25` | 순수 Python, numpy만 의존(이미 설치), 260문서 ~1ms |
| 형태소 분석 | `kiwipiepy` | C 확장 wheel 내장, JVM/MeCab 불필요, 폐쇄망 친화적 |

### 3.3 필요 패키지 (폐쇄망 반입)

| 패키지 | 크기 | 의존성 |
|--------|------|--------|
| `rank-bm25` 0.2.2 | ~10KB | numpy (이미 설치) |
| `kiwipiepy` 0.18.x | ~30MB (wheel) | 없음 |

```bash
# 인터넷 환경에서 다운로드
pip download rank-bm25 kiwipiepy --dest ./wheels --only-binary=:all:

# 폐쇄망에서 설치
pip install --no-index --find-links=./wheels rank-bm25 kiwipiepy
```

### 3.4 구현 설계

#### 3.4-A. 토크나이저 유틸리티 (신규: `backend/services/korean_tokenizer.py`)

- kiwipiepy 래퍼, 조사/어미/기호 제외한 의미 토큰 추출
- **kiwipiepy 미설치 시 공백 기반 폴백** → 기존 동작과 동일 (무중단)
- 싱글턴 패턴으로 Kiwi 인스턴스 재활용

#### 3.4-B. BM25 키워드 검색 (재작성: `backend/services/keyword_search.py`)

- `BM25Okapi`로 인덱스 구축 (제목 토큰 3× 가중)
- search-index.json 로드 시 lazy 토크나이즈 + 캐시
- `reload_bm25_index()` 함수로 인덱스 재생성 시 캐시 무효화
- **rank-bm25 미설치 시 나이브 검색 폴백** (현재 로직 보존)

#### 3.4-C. 하이브리드 검색 (`backend/services/vector_search.py`)

- **코드 변경 불필요** — `hybrid_search()`는 키워드 결과를 RRF 순위로만 사용하므로 BM25 점수 스케일 무관

#### 3.4-D. 인덱스 빌드 (`tools/build-search-index.py`)

- **변경 불필요** — search-index.json 포맷 동일, BM25는 런타임 구축

### 3.5 영향 분석

| 항목 | 영향 | 대응 |
|------|------|------|
| 키워드 검색 결과 순위 | BM25 ≠ 나이브 → RRF 최종 결과 변동 가능 | 기존 15문항 Before/After 비교 |
| 첫 요청 지연 | BM25 인덱스 구축 ~1-2초 | lazy-load, 이후 캐시 |
| vector_search | 변경 없음 | - |
| chat.py | 변경 없음 | - |
| 프론트엔드 | 변경 없음 | - |

### 3.6 위험도: **중간**

검색 결과 순서가 바뀔 수 있으나, 이중 폴백(rank-bm25 미설치 → 나이브, kiwipiepy 미설치 → 공백)으로 장애 전파 차단.

### 3.7 롤백

- **즉시**: `keyword_search.py`를 git checkout으로 원복
- **자동**: 패키지 미설치 시 나이브 검색으로 자동 폴백

## 4. Step 3: 답변 검증 (경량 할루시네이션 체크)

### 4.1 목표

LLM 응답 생성 후 경량 검증을 수행하여, 문서에 뒷받침되지 않는 답변에 면책 문구를 추가한다.

### 4.2 설계 판단

| 접근법 | 장점 | 단점 | 선정 |
|--------|------|------|:----:|
| LLM 자기 검증 | 추가 인프라 불필요 | 자기 확인 편향 | **선정** |
| NLI cross-encoder | 정밀한 함의 판단 | 추가 모델 ~1GB VRAM | 향후 검토 |

- 기본 **비활성화** (`ANSWER_VERIFICATION_ENABLED = False`)
- 관리자 설정에서 토글 가능
- 검증 실패 시 항상 SUPPORTED 반환 (답변 차단 없음)

### 4.3 구현 설계

#### 4.3-A. 검증 서비스 (신규: `backend/services/answer_verifier.py`)

- 검증 프롬프트로 LLM에 질의: SUPPORTED / PARTIAL / UNSUPPORTED
- PARTIAL → 면책 문구: "⚠ 일부는 제공된 문서에서 직접 확인되지 않았습니다"
- UNSUPPORTED → 면책 문구: "⚠ 충분한 근거를 찾지 못했습니다"
- 컨텍스트/답변을 요약하여 토큰 절약 (컨텍스트 상위 3건 × 200자, 답변 500자)

#### 4.3-B. 동기 응답 통합 (`backend/api/chat.py`)

- `chat()` 엔드포인트에서 LLM 응답 생성 후 `verify_answer()` 호출
- 면책 문구가 있으면 답변에 append

#### 4.3-C. 스트리밍 응답 통합 (`backend/api/chat.py`)

- 스트리밍 완료 후 검증 → 면책 문구를 추가 토큰으로 전송
- `done` 페이로드에 `verification` 필드 추가

#### 4.3-D. 설정 (`backend/config.py`)

```python
ANSWER_VERIFICATION_ENABLED = False  # 기본 비활성화
```

### 4.4 영향 분석

| 항목 | 영향 |
|------|------|
| 응답 시간 | 활성화 시 +2~5초 (LLM 추가 호출 1회) |
| 기존 답변 | 변화 없음 (면책 문구만 조건부 추가) |
| 스트리밍 UX | 완료 후 면책 문구가 추가 표시 |
| 비활성화 시 | 완전히 무영향 (코드 경로 스킵) |

### 4.5 한계

- **자기 확인 편향**: 동일 모델로 생성+검증 → 감지 못할 수 있음
- **지연 증가**: 모든 응답에 추가 LLM 호출
- **gemma3:4b NLI 능력 제한**: 명백한 할루시네이션만 감지 가능

### 4.6 위험도: **낮음**

기본 비활성화. 롤백은 config 값 변경으로 즉시 가능.

## 7. 회귀 테스트 계획

### 7.1 Step 1 검증 (언어 제어)

| # | 시나리오 | 검증 포인트 |
|:-:|----------|------------|
| L1 | "rag 성능 측정 기준같은게 있어?" → "결과표값 알려줘" | 재작성 쿼리가 한국어인지, 답변이 한국어인지 |
| L2 | "FY 2020 defense budget?" | 영어 질문이어도 한국어 답변 |
| L3 | REASON 라우트 질문 | Agent 검색 쿼리가 한국어인지 (백엔드 로그) |

### 7.2 Step 2 검증 (BM25)

기존 15문항 + 형태소 분석 특화 질문으로 Before/After 비교:

| # | 쿼리 | 현재 키워드 검색 | 기대 BM25 결과 |
|:-:|------|:---:|:---:|
| K1 | "비행시험을 수행하는 절차" | "비행시험을" 불일치 | "비행시험"+"수행"+"절차" 분리 매칭 |
| K2 | "전투기 최대 속도" | substring만 | "전투기" IDF 높은 가중치 |
| K3 | "AESA 레이더 탐지거리" | 영어 약어 매칭 | 동일 (영어 약어 보존) |

**필수 확인**: Recall@5 ≥ 100%, MRR ≥ 0.967 유지

### 7.3 Step 3 검증 (답변 검증)

| # | 시나리오 | 기대 검증 결과 |
|:-:|----------|---------------|
| V1 | 명확한 사실 질문 | SUPPORTED |
| V2 | 문서에 없는 질문: "KF-21 가격은?" | UNSUPPORTED + 면책 문구 |

## 8. 현장 테스트 가이드

### 8.1 사전 준비

1. **패키지 설치 확인** (폐쇄망인 경우 wheel 반입 필요)
   ```bash
   pip install rank-bm25 kiwipiepy
   # 또는 폐쇄망:
   pip install --no-index --find-links=./wheels rank-bm25 kiwipiepy
   ```
2. **백엔드 재시작** — 프롬프트 변경 + 신규 모듈 반영을 위해 필수
   ```bash
   cd backend && python main.py
   ```
3. 패키지 미설치 시에도 서버는 정상 기동됨 (나이브 검색 폴백)

### 8.2 Step 1 검증: 언어 제어

> **핵심**: 후속 질문에서 영어 전환이 발생하지 않는지 확인

| # | 테스트 질문 | 예상 결과 | 확인 포인트 |
|:-:|-----------|----------|------------|
| L1 | "RAG 성능 측정 기준같은게 있어?" | Recall@5, MRR 등 용어 포함 한국어 답변 | 영어 용어는 보존, 설명은 한국어 |
| L2 | *(L1 대화에 이어서)* "관련해서 결과표값 알려줘" | 한국어 답변 + 수치 표 | **이전에는 여기서 영어로 전환됨** → 이제 한국어 유지 |
| L3 | "FY 2020 defense budget?" | 한국어 답변 | 영어 질문이어도 한국어로 응답 |
| L4 | "KF-21의 AESA 레이더와 기존 방식을 비교 분석해줘" | 한국어 분석 답변 | REASON 라우트 → Agent 쿼리도 한국어 |

**L2가 가장 중요한 테스트입니다** — 이전 세션에서 실제로 영어 전환이 발생했던 시나리오.

### 8.3 Step 2 검증: BM25 + 형태소 분석

> **핵심**: 조사/어미 변형에 강건해졌는지, 기존 검색 품질이 유지되는지 확인

| # | 테스트 질문 | 이전 (나이브) | 현재 (BM25) | 확인 포인트 |
|:-:|-----------|:---:|:---:|------------|
| K1 | "비행시험을 수행하는 절차는?" | "비행시험을" 그대로 매칭 | "비행"+"시험"+"수행"+"절차" 분리 | 더 다양한 문서 검색 |
| K2 | "전자전체계 구성요소" | "전자전체계" 통째로 매칭 | "전자전"+"체계" 분리 | 복합명사 분리 효과 |
| K3 | "AESA 레이더 탐지거리" | 1건만 검색 | 5건 검색 | **가장 큰 개선** — "탐지"+"거리" 분리 |
| K4 | "KF-21 전투기 소개" | 5건 | 5건 | 퇴행 없음 확인 |
| K5 | "감항인증을 받기 위한 기준" | "감항인증을" 매칭 | "감항"+"인증"+"기준" 분리 | 조사 제거 효과 |

**K3, K5가 핵심 개선 사례** — 이전에는 못 찾거나 적게 찾던 결과가 늘어남.

### 8.4 Step 3 검증: 답변 검증

> **핵심**: 기본 비활성화 상태에서 영향 없음 확인, 활성화 시 면책 문구 동작 확인

**비활성화 상태 (기본값)**
- 아무 질문이나 해봐도 답변 끝에 면책 문구(`---` 구분선 + 이탤릭 문구)가 없어야 함
- 기존과 완전히 동일한 동작

**활성화 방법**: 관리자 설정 → AI 탭 → `answer_verification_enabled` → `true`로 변경

| # | 테스트 질문 | 예상 결과 | 확인 포인트 |
|:-:|-----------|----------|------------|
| V1 | "KF-21 전투기 개발 배경을 알려줘" | 한국어 답변, 면책 없음 | 문서 근거 충분 → SUPPORTED |
| V2 | "KF-21 한 대 가격이 정확히 얼마야?" | "찾지 못했습니다" 또는 답변 + 면책 문구 | 문서에 가격 정보 없음 |
| V3 | "KF-21과 F-35 최대 속도를 숫자로 비교해줘" | 답변 + 면책 문구 가능 | 구체 수치가 문서에 없으면 면책 |

**주의**: gemma3:4b의 NLI 능력 한계로 모호한 일반 상식("비용이 높습니다" 등)은 감지하지 못할 수 있음. 구체적인 거짓 수치("마하 1.81", "740km")는 감지 가능.

### 8.5 기존 기능 퇴행 체크리스트

| # | 확인 항목 | 방법 |
|:-:|----------|------|
| R1 | 일반 질문 정상 답변 | "KF-21 전투기 소개" → 정상 답변 |
| R2 | 검색 기능 정상 | Explorer 검색창에서 키워드 검색 |
