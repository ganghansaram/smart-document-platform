## 1. 개요

### 배경

Phase 8(안정성 개선)까지 완료된 상태에서, 사용자 경험(UX) 측면의 업그레이드 3건을 진행한다.

### 요구사항 요약

| # | 항목 | 핵심 내용 | 영향 범위 |
|---|------|----------|----------|
| 1 | 번역 시스템 검색 | 헤더 Search 버튼 → 업로드 문서 본문/메모 검색 | translator.html, js/translator.js, 백엔드 API 추가 |
| 2 | 번역 시스템 홈 배너 | 카드 목록 상단에 피처 소개 카드 배너 | translator.html (view-list 내부) |
| 3 | 로그인 화면 개선 | 좌우 분할 (좌=폼, 우=스크린샷 슬라이드) | login.html 단독 |

### 참조 문서

- `memory/theme-guide.md` — UI 테마 기준서 (색상, 타이포, 스페이싱, radius, shadow, z-index, 컴포넌트 규격)
- `css/tokens.css` — 디자인 토큰 변수

> 모든 기능에서 새 UI 컴포넌트 작성 시 반드시 기준서 값을 사용할 것. 기능 1의 검색 오버레이는 Explorer 패턴을 기반으로 하되, 토큰 변수 적용 확인.

### 작업 우선순위

```
1순위: 번역 시스템 검색 ─── 기능적 가치 최고, 문서 증가 시 필수
2순위: 번역 시스템 홈 배너 ─ 검색 버튼 추가와 함께 홈 화면 정비
3순위: 로그인 화면 개선 ──── 시각적 완성도, 콘텐츠 소재 의존
```

## 2. 기능 1: 번역 시스템 검색

### 2.1 현재 상태

- Explorer에는 하이브리드 검색 (키워드 + FAISS 벡터) 존재
- Translator에는 검색 기능 없음
- 문서가 늘어나면 원하는 내용을 찾기 어려운 상황

### 2.2 설계 방향

#### 검색 범위

| 대상 | 소스 | 비고 |
|------|------|------|
| 문서 본문 (원문) | PDF 텍스트 (PyMuPDF 추출) | 업로드 시 자동 인덱싱 |
| 메모 | annotations.json의 memo 필드 | 실시간 검색 (파일 직접 조회) |
| 문서 제목 | meta.json의 title 필드 | 부가 정보 |

> **제외**: 번역된 텍스트 (우측 패널) — 원문 검색만으로 충분

#### 검색 엔진: 키워드 검색만 (초기)

- 개인 문서 규모 (수십~수백 건)에서 벡터 검색은 과잉
- Explorer의 keyword_search.py 패턴을 참고하되, 유저별 독립 인덱스
- 추후 문서량 증가 시 벡터 검색 추가 가능 (확장 포인트 확보)

#### 인덱스 구조

```
data/translator/{username}/
├── _index.json        (기존 — 문서 목록)
├── _folders.json      (기존 — 폴더)
├── _search_index.json (신규 — 전문 검색 인덱스)
└── {doc_id}/
    ├── meta.json
    ├── annotations.json
    └── ...
```

**_search_index.json 형식** (유저 단위 통합):
```json
{
  "version": 1,
  "updated_at": "2026-03-09T...",
  "documents": {
    "{doc_id}": {
      "title": "논문제목.pdf",
      "pages": {
        "1": "First page extracted text...",
        "2": "Second page extracted text..."
      }
    }
  }
}
```

> **대안 검토**: 문서별 인덱스 vs 유저별 통합 인덱스
> - 문서별: 파일 I/O 분산, 문서 삭제 시 파일만 삭제하면 됨
> - 유저별 통합: 검색 시 파일 1개만 로드, 구현 단순
> - **결정**: 유저별 통합 채택 (문서 수가 적으므로 단일 파일 로드가 효율적)

#### 인덱싱 방식: 업로드 시 자동 (수동 불필요)

Explorer는 관리자가 조직 공유 문서를 관리하므로 수동 인덱싱 옵션이 있지만,
Translator는 **개인이 자기 문서를 올리고 자기가 검색**하는 구조.

| Explorer (조직 공유) | Translator (개인 작업) |
|---------------------|----------------------|
| 관리자 설정에 자동/수동 토글 | 항상 자동 |
| 벡터 인덱싱 포함 (시간 소요) | 키워드만 (PyMuPDF 추출 1~2초) |
| 별도 reindex API | 불필요 |

- PDF 텍스트 추출(PyMuPDF)은 수백 페이지도 1~2초 → 업로드 체감 지연 거의 없음
- 사용자에게 "인덱싱" 개념을 노출하지 않음 → 올리면 바로 검색 가능

| 이벤트 | 동작 |
|--------|------|
| PDF 업로드 | PyMuPDF로 전체 페이지 텍스트 추출 → `_search_index.json`에 추가 |
| 문서 삭제 | `_search_index.json`에서 해당 doc_id 제거 |
| 문서 제목 변경 | `_search_index.json`의 title 갱신 |

#### 기존 문서 마이그레이션

이미 업로드된 문서(`_search_index.json` 없는 유저)는 **검색 API 첫 호출 시 온디맨드 빌드**.
- 유저가 검색 버튼 누름 → `_search_index.json` 없음 → 해당 유저 문서 일괄 인덱싱 → 결과 반환
- 최초 1회만 약간의 지연 (문서 수에 비례), 이후는 즉시

#### 검색 위치

**헤더 Search 버튼 → 홈(카드목록)과 뷰어 모두에서 동일하게 동작**
- 헤더는 양쪽 화면에서 항상 노출
- 검색 결과 클릭 시: 홈이면 문서 열기+페이지 이동, 뷰어면 페이지 이동(같은 문서) 또는 문서 전환(다른 문서)

#### 백엔드 API

```
GET /api/translator/search?q={query}
```

응답:
```json
{
  "results": {
    "memos": [
      {
        "doc_id": "...",
        "doc_title": "논문.pdf",
        "page": 3,
        "memo": "이 부분 중요 — 열처리 조건",
        "highlight_id": "h_...",
        "color": "yellow",
        "text": "highlighted text",
        "snippet": "...context around match..."
      }
    ],
    "pages": [
      {
        "doc_id": "...",
        "doc_title": "논문.pdf",
        "page": 7,
        "snippet": "...context around match...",
        "score": 5
      }
    ]
  },
  "query": "열처리",
  "total": 12
}
```

#### 프론트엔드 UI

- **진입**: 헤더에 Search 버튼 추가 (Explorer와 동일 위치)
- **위젯**: Explorer의 search-overlay 패턴 재활용
  - 오버레이 + 입력창 + 결과 영역
  - 300ms 디바운스
- **결과 표시**: 두 그룹으로 분리
  ```
  ┌─ 검색: "열처리" ──────────────────────┐
  │                                       │
  │  메모 (3건)                            │
  │  ├─ [논문A.pdf] p.3 — "열처리 조건..." │
  │  ├─ [논문B.pdf] p.12 — "열처리 온도..."│
  │  └─ [논문C.pdf] p.1 — "열처리 공정..." │
  │                                       │
  │  본문 (9건)                            │
  │  ├─ [논문A.pdf] p.5 — "...열처리..."   │
  │  ├─ [논문A.pdf] p.8 — "...열처리..."   │
  │  └─ ...                               │
  └───────────────────────────────────────┘
  ```
- **결과 클릭 동작**:
  1. 해당 문서가 현재 열려있지 않으면 → 문서 열기 (openDocument)
  2. 해당 페이지로 이동 (goToPage)
  3. 검색 오버레이 닫기

### 2.3 구현 단계

```
Step 1: 백엔드 — 인덱싱 로직
  - upload_pdf()에 텍스트 추출 + _search_index.json 저장 추가
  - delete_document()에 인덱스 정리 추가
  - rename_document()에 title 갱신 추가

Step 2: 백엔드 — 검색 API
  - GET /api/translator/search?q=...
  - _search_index.json 없으면 온디맨드 빌드 (기존 문서 마이그레이션)
  - _search_index.json + annotations.json 동시 검색
  - 결과를 memos/pages 그룹으로 반환

Step 3: 프론트엔드 — 검색 UI
  - 헤더 Search 버튼 추가
  - 검색 오버레이 (CSS + HTML)
  - 검색 로직 (fetch → 결과 렌더링)
  - 결과 클릭 → 문서 열기 + 페이지 이동
```

## 3. 기능 2: 번역 시스템 홈 배너

### 3.1 현재 상태

```
┌─ translator 홈 ──────────────────┐
│  [업로드 존 — 드래그 앤 드롭]      │
│  [카드 그리드 — 문서 목록]         │
│  [푸터]                          │
└──────────────────────────────────┘
```

### 3.2 목표

```
┌─ translator 홈 ──────────────────────────────────────┐
│  ┌──────────┬──────────┬──────────┬──────────┐       │
│  │ 업로드    │ 번역     │ 마킹     │ AI       │ ← 신규│
│  │ 드래그로  │ 페이지별  │ 형광펜+  │ 드래그로  │       │
│  │ 간편 업로드│ 선택 번역 │ 메모 정리 │ 즉시 분석 │       │
│  └──────────┴──────────┴──────────┴──────────┘       │
│  [업로드 존 — 드래그 앤 드롭]                           │
│  [카드 그리드 — 문서 목록]                              │
│  [푸터]                                              │
└──────────────────────────────────────────────────────┘
```

### 3.3 배너 설계: 카드형 피처 소개

슬라이드 대신 **카드형 가로 나열** 채택.

| 비교 | 자동 슬라이드 | 카드형 가로 나열 |
|------|-------------|----------------|
| 정보 전달 | 한 번에 1개, 기다려야 함 | 한눈에 4개 모두 확인 |
| 사용자 반응 | 자동 넘김 싫어하는 트렌드 | 정적, 스캔하기 쉬움 |
| 구현 복잡도 | JS 타이머 + 전환 로직 | CSS만으로 충분 |
| 트렌드 | 2020년대 초반 | 2024~2026 현재 주류 |

#### 카드 구성

| # | 아이콘 | 제목 | 설명 (1줄) |
|---|--------|------|-----------|
| 1 | 문서 아이콘 | PDF 업로드 | 드래그 앤 드롭으로 간편 업로드 |
| 2 | 번역 아이콘 | 페이지별 번역 | 원하는 페이지만 선택하여 번역 |
| 3 | 형광펜 아이콘 | 마킹 & 메모 | 형광펜 표시 + 메모로 핵심 정리 |
| 4 | AI 아이콘 | AI 텍스트 분석 | 드래그 선택으로 즉시 번역/요약 |

#### 비주얼 스타일

- 반투명 카드 (글래스모피즘 또는 미니멀 보더)
- SVG 아이콘 (외부 의존 없음, 폐쇄망 호환)
- 다크/라이트 모드 대응 (tokens.css 변수 활용)
- 접기/닫기 버튼 + `localStorage('translator-banner-collapsed')` 상태 저장

#### 기술 구현

- **CSS only** (JS 불필요) — flexbox 4칸 + gap
- `translator.html`의 `view-list` 내부, `upload-zone` 위에 삽입
- `css/translator.css`에 배너 스타일 추가

### 3.4 소재

- 스크린샷 없이 **아이콘 + 텍스트만으로 구성** (소재 의존 제거)
- 추후 스크린샷을 카드 배경으로 넣는 것은 선택사항

## 4. 기능 3: 로그인 화면 개선

### 4.1 현재 상태

```
┌──────────────────────────────────┐
│         (어두운 배경)              │
│                                  │
│       [KAI 로고]                 │
│       Smart Knowledge Platform   │
│                                  │
│       ┌──────────────┐           │
│       │  로그인 카드   │           │
│       │  ID / PW     │           │
│       │  [로그인]     │           │
│       └──────────────┘           │
│                                  │
│       [공지사항]                  │
│       [저작권]                    │
└──────────────────────────────────┘
```

### 4.2 목표: 좌우 분할 + 우측 Hero 스크린샷 슬라이드

```
┌────────────────────┬─────────────────────────┐
│                    │                         │
│   [KAI 로고]       │   ┌─────────────────┐   │
│   Smart Knowledge  │   │                 │   │
│   Platform         │   │  스크린샷 이미지   │   │
│                    │   │  + 오버레이 카피   │   │
│   ┌────────────┐   │   │                 │   │
│   │ ID         │   │   │  "페이지별로     │   │
│   │ PW         │   │   │   번역하고      │   │
│   │ [로그인]    │   │   │   메모하세요"    │   │
│   └────────────┘   │   │                 │   │
│                    │   └─────────────────┘   │
│   [공지사항]       │   · · · (인디케이터)      │
│   [저작권]         │                         │
│                    │                         │
└────────────────────┴─────────────────────────┘
 (< 900px 이하: 우측 패널 숨김, 기존 중앙 레이아웃 폴백)
```

### 4.3 설계 상세

#### 레이아웃

- 좌측 (약 45%): 기존 로그인 폼 (로고 + 카드 + 공지 + 저작권)
- 우측 (약 55%): 특징 소개 패널
- 반응형: 화면 폭 < 900px → 우측 패널 숨김 (기존 중앙 레이아웃 폴백)

#### 우측 패널 비주얼: Hero 스크린샷 + 오버레이 텍스트

- **실제 UI 스크린샷** 위에 반투명 그라디언트 + 한 줄 카피
- 자동 fade 전환 (6초 간격)
- 하단 도트 인디케이터

| # | 스크린샷 대상 | 카피 |
|---|-------------|------|
| 1 | Explorer 문서 탐색 화면 | "수천 페이지 기술 문서를 섹션 단위로 탐색" |
| 2 | Explorer AI 채팅 | "AI에게 질문하면 관련 내용을 찾아드립니다" |
| 3 | Translator 듀얼 뷰어 | "영문 논문을 페이지별로 한국어 번역" |
| 4 | Translator 마킹/메모 | "중요 부분을 형광펜으로 표시하고 메모" |

#### 기술 구현

- `login.html` 내부 CSS/HTML 변경 (JS 로직 변경 없음, 슬라이드 JS 최소 추가)
- CSS Grid 2칸 분할
- 기존 어두운 배경 + breathing 효과 유지
- 우측 패널: 어두운 톤, 스크린샷은 둥근 모서리 + 미세 그림자

#### 소재 제작

- Playwright MCP로 실제 화면 캡처 → `css/images/login-slide-*.png`
- 또는 수동 스크린샷 촬영
- 이미지 4장 (각 1280x800 정도, WebP/PNG)

## 5. 작업 순서

```
Phase A: 번역 시스템 검색 (기능 1) ✅ 완료
  ├─ Step 1: 백엔드 — 업로드 시 자동 인덱싱 + 삭제/제목변경 연동 ✅
  ├─ Step 2: 백엔드 — 검색 API (온디맨드 마이그레이션 포함) ✅
  └─ Step 3: 프론트엔드 — 헤더 Search 버튼 + 오버레이 + 결과 네비게이션 ✅

Phase B: 번역 시스템 홈 배너 (기능 2) ✅ 완료 (형태 변경)
```
