# 지식 그래프 (Knowledge Graph) 구현 계획서

> **작성일**: 2026-03-10
> **상태**: 계획 (미착수)
> **목적**: Explorer(조직 표준문서)와 Translator(개인 문서)에 문서 간 관계를 시각화하는 지식 그래프 뷰 구현

---

## 1. 목표

- 문서 간 관계를 **force-directed 그래프**로 시각화
- Explorer: 조직 표준문서 간 유사도 · 용어 공유 · 계층 관계 자동 연결
- Translator: 개인 PDF 간 마킹/메모 키워드 · 폴더 관계 · 텍스트 유사도 연결
- 노드 클릭 → 해당 문서로 이동하는 **탐색 진입점** 역할
- **별도 전체 화면 뷰**로 구현하여 향후 필터/클러스터/타임라인 등 확장 가능

---

## 2. 현황 분석

### 2-1. Explorer — 사용 가능한 데이터

| 데이터 | 파일 | 규모 | 그래프 활용 |
|--------|------|------|------------|
| 검색 인덱스 | `data/search-index.json` | 260개 섹션, 735KB | 노드 (문서/섹션), 텍스트 비교용 |
| 벡터 인덱스 | `data/vector-index.faiss` | 260개, 1024차원, 1MB | **문서 간 유사도 엣지** (L2 거리) |
| 벡터 메타 | `data/vector-index_meta.json` | 260개, 236KB | 노드 메타데이터 |
| 용어집 | `data/glossary.json` | 26,048개 용어, 2.4MB | **용어 공유 엣지** |
| 메뉴 구조 | `data/menu.json` | ~178개 노드, 5.5KB | **계층 엣지** (부모-자식) |

**핵심**: 벡터 임베딩이 이미 계산·저장되어 있어 문서 간 유사도를 **추가 연산 없이** FAISS로 즉시 계산 가능.

```
유사도 = 1 / (1 + L2_distance)
임계값 = MIN_VECTOR_SCORE (0.48, config.py)
```

### 2-2. Translator — 사용 가능한 데이터

| 데이터 | 파일 | 규모 | 그래프 활용 |
|--------|------|------|------------|
| 문서 목록 | `_index.json` | 사용자별 N개 | 노드 |
| 페이지 텍스트 | `_search_index.json` | 페이지별 전문, ~755KB (12문서) | 키워드 추출, 유사도 |
| 마킹/메모 | `{doc_id}/annotations.json` | 문서별 N개 | **메모 키워드 공유 엣지** |
| 폴더 구조 | `_folders.json` | 사용자별 N개 | **폴더 관계 엣지** |
| 문서 메타 | `{doc_id}/meta.json` | 문서별 | 노드 메타 (제목, 페이지 수, 업로드 시간) |

**핵심**: 벡터 인덱스가 **없음**. 키워드 매칭과 메모/폴더 기반 관계가 주력. 벡터 유사도를 원하면 임베딩 신규 계산 필요.

### 2-3. Explorer vs Translator 비교

| 항목 | Explorer | Translator |
|------|----------|------------|
| 문서 성격 | 조직 공유 표준문서 | 개인 PDF (논문/규격서) |
| 문서 수 | ~260 섹션 (고정적) | 사용자별 수십 건 (유동적) |
| 벡터 임베딩 | ✅ FAISS에 저장됨 | ❌ 없음 (신규 계산 필요) |
| 자동 유사도 | ✅ 즉시 계산 가능 | ⚠️ 키워드 기반 또는 신규 임베딩 |
| 사용자 생성 관계 | ❌ 없음 | ✅ 마킹/메모/폴더 |
| 계층 구조 | ✅ menu.json | ✅ _folders.json |
| 용어 연결 | ✅ 26,048 용어집 | ❌ 없음 |

## 3. 시각화 라이브러리 선정

### 3-1. 후보 비교

| 라이브러리 | 번들 크기 | 의존성 | ES5 | 렌더링 | 폐쇄망 |
|-----------|----------|--------|-----|--------|--------|
| **D3.js (d3-force)** | ~250KB | 없음 (UMD) | ✅ | SVG/Canvas | ✅ |
| **VivaGraphJS** | ~50KB | **Zero** | ✅ | WebGL/SVG/Canvas | ✅ |
| Force-Graph | ~100KB+ | d3-force | ⚠️ 확인 필요 | Canvas | ✅ |
| Cytoscape.js | ~300KB | 없음 | ✅ | Canvas | ✅ |
| Sigma.js | ~200KB | graphology | ⚠️ ESM | WebGL | ⚠️ |

### 3-2. 권장: D3.js (d3-force)

**선정 이유**:
- 업계 표준, 문서·예제 풍부 — 유지보수 용이
- UMD 번들 단일 파일 (`d3.v7.min.js`) → `js/lib/d3/`에 배치
- Canvas 렌더링 가능 (500개 노드까지 원활)
- 이미 프로젝트에 PDF.js, Monaco Editor 등 대형 라이브러리 운용 중 — 250KB 추가 부담 미미
- force-directed 레이아웃 품질이 검증됨

**대안**: 문서 수가 적은 Translator에는 VivaGraphJS(50KB, zero-dep)도 고려 가능. 단, 두 시스템에 같은 라이브러리를 쓰는 것이 유지보수에 유리.

### 3-3. 배치 계획

```
js/lib/d3/
└── d3.v7.min.js    (250KB, UMD 번들)
```

- CDN 없이 로컬 파일로 로드
- `<script src="js/lib/d3/d3.v7.min.js"></script>`
- Explorer와 Translator 모두 동일 파일 참조

## 4. 그래프 데이터 모델

### 4-1. 공통 노드/엣지 스키마

```json
{
  "nodes": [
    {
      "id": "unique-node-id",
      "label": "문서/섹션 제목",
      "type": "document | section | folder | glossary",
      "group": "카테고리 또는 폴더명",
      "size": 10,
      "meta": { ... }
    }
  ],
  "edges": [
    {
      "source": "node-id-1",
      "target": "node-id-2",
      "type": "similarity | keyword | hierarchy | folder | glossary",
      "weight": 0.75,
      "label": "연결 설명 (선택)"
    }
  ]
}
```

### 4-2. Explorer 엣지 유형

| 엣지 유형 | 소스 | 생성 방식 | 가중치 |
|-----------|------|----------|--------|
| **벡터 유사도** | FAISS 인덱스 | 문서 간 L2 거리 → 유사도 변환 | 0.48~1.0 |
| **용어집 공유** | glossary.json + search-index | 동일 약어가 등장하는 문서 쌍 | 공유 용어 수 |
| **메뉴 계층** | menu.json | 부모-자식 관계 | 1.0 (고정) |
| **동일 카테고리** | menu.json path | 같은 상위 카테고리 소속 | 1.0 (고정) |

**엣지 생성 우선순위**: 벡터 유사도 > 용어집 공유 > 메뉴 계층

### 4-3. Translator 엣지 유형

| 엣지 유형 | 소스 | 생성 방식 | 가중치 |
|-----------|------|----------|--------|
| **메모 키워드 공유** | annotations.json | 메모 텍스트에서 키워드 추출 → 공출현 | 공유 키워드 수 |
| **마킹 텍스트 공유** | annotations.json | 마킹된 원문 텍스트의 키워드 공출현 | 공유 키워드 수 |
| **같은 폴더** | _folders.json | 동일 folder_id 소속 | 1.0 (고정) |
| **제목 유사도** | _index.json | 제목 토큰 공유 | Jaccard 유사도 |
| **페이지 텍스트 유사도** | _search_index.json | TF-IDF 또는 임베딩 유사도 | 0~1.0 |

**엣지 생성 우선순위**: 메모 키워드 > 마킹 텍스트 > 같은 폴더 > 텍스트 유사도

## 5. 백엔드 API 설계

### 5-1. Explorer 그래프 API

```
GET /api/graph/explorer
```

**응답**:
```json
{
  "nodes": [...],
  "edges": [...],
  "stats": {
    "node_count": 42,
    "edge_count": 128,
    "cluster_count": 6
  }
}
```

**처리 흐름**:
1. `search-index.json` 로드 → 노드 생성 (문서 URL 기준 그룹화)
2. FAISS 인덱스 로드 → 전체 벡터 간 pairwise 유사도 계산
3. 임계값(0.48) 이상인 쌍 → 유사도 엣지 생성
4. `menu.json` 파싱 → 계층 엣지 생성
5. (선택) 용어집 매칭 → 용어 공유 엣지 생성
6. 결과 캐싱 (`data/graph-cache-explorer.json`)

**캐싱 전략**:
- 검색 인덱스 변경 시(업로드/재인덱싱) 캐시 무효화
- 캐시 유효 시 파일에서 직접 반환 (FAISS 연산 스킵)

### 5-2. Translator 그래프 API

```
GET /api/graph/translator
```

**응답**: 동일 스키마 (nodes + edges + stats)

**처리 흐름**:
1. `_index.json` 로드 → 문서 노드 생성
2. `_folders.json` 로드 → 폴더 노드 + 폴더 엣지 생성
3. 각 문서의 `annotations.json` 로드 → 키워드 추출
4. 문서 간 키워드 공출현 → 메모/마킹 엣지 생성
5. `_search_index.json` 로드 → 제목 유사도 엣지 생성
6. (선택) 페이지 텍스트 → TF-IDF 유사도 엣지
7. 결과를 `_graph_cache.json`에 저장

**캐싱 전략**:
- 문서 업로드/삭제/마킹 변경 시 캐시 무효화
- 사용자별 독립 캐시 (`data/translator/{username}/_graph_cache.json`)

### 5-3. 신규 백엔드 파일

```
backend/api/graph.py          ← API 라우터 (2개 엔드포인트)
backend/services/graph_service.py  ← 그래프 생성 로직
```

## 6. 프론트엔드 설계

### 6-1. 진입 방식

| 시스템 | 진입 | 동작 |
|--------|------|------|
| Explorer | 헤더 네비게이션에 **Graph** 버튼 추가 | 본문 영역을 그래프 뷰로 전환 (문서 뷰와 토글) |
| Translator | 문서 목록 뷰 상단에 **Graph** 버튼 추가 | 목록 뷰를 그래프 뷰로 전환 |

- 그래프 뷰 ↔ 기존 뷰 토글 (뒤로가기 시 이전 상태 복원)
- 그래프에서 노드 클릭 → 해당 문서로 이동 (뷰 전환)

### 6-2. 화면 레이아웃

```
┌─ 공유 헤더 ───────────────────────────────────────────┐
├─ 필터 바 ─────────────────────────────────────────────┤
│ [카테고리▾] [엣지 유형▾] [연결 강도▾] [검색_________]  │
├───────────────────────────────────┬────────────────────┤
│                                   │  노드 상세 패널     │
│                                   │  (280px, 접기 가능) │
│      Force-directed Graph         │                    │
│      (Canvas 렌더링)              │  제목: ○○○         │
│                                   │  카테고리: ○○○     │
│      ○───○                       │  연결: N개          │
│     / \   \                      │                    │
│    ○   ○───○                     │  관련 문서:         │
│                                   │  · 문서A (0.87)    │
│                                   │  · 문서B (0.72)    │
│                                   │                    │
│   줌: 스크롤 / 핀치               │  [문서 열기]        │
│   팬: 드래그                      │                    │
├───────────────────────────────────┴────────────────────┤
│ 노드 N개 · 엣지 N개 · 클러스터 N개         [← 목록으로] │
└────────────────────────────────────────────────────────┘
```

### 6-3. 인터랙션

| 동작 | 효과 |
|------|------|
| 노드 호버 | 노드 강조 + 연결된 엣지 강조 + 나머지 투명도 낮춤 |
| 노드 클릭 | 우측 상세 패널에 정보 표시 |
| 노드 더블클릭 | 해당 문서로 이동 (그래프 뷰 → 문서 뷰) |
| 노드 드래그 | 노드 위치 수동 조정 (force 시뮬레이션 일시 정지) |
| 배경 드래그 | 그래프 팬 |
| 스크롤 | 줌 인/아웃 |
| 필터 변경 | 엣지 유형/강도 필터링 → 그래프 재배치 |
| 검색 | 매칭 노드 강조 + 중앙 이동 |

### 6-4. 시각 표현

| 요소 | Explorer | Translator |
|------|----------|------------|
| 노드 크기 | 연결 수(degree) 비례 | 마킹/메모 수 비례 |
| 노드 색상 | 카테고리별 (설계=파랑, 시험=초록 등) | 폴더별 색상 |
| 엣지 굵기 | 유사도 가중치 비례 | 공유 키워드 수 비례 |
| 엣지 스타일 | 실선(유사도), 점선(계층), 파선(용어) | 실선(키워드), 점선(폴더) |
| 클러스터 | 카테고리별 자동 그룹핑 | 폴더별 그룹핑 |

### 6-5. 신규 프론트엔드 파일

```
js/knowledge-graph.js     ← 그래프 렌더링 + 인터랙션 (공통 모듈)
css/knowledge-graph.css   ← 그래프 뷰 스타일
js/lib/d3/d3.v7.min.js   ← D3.js 라이브러리
```

- `knowledge-graph.js`는 Explorer와 Translator 모두에서 사용
- 초기화 시 시스템 타입(`explorer` | `translator`)을 파라미터로 전달
- API 엔드포인트, 노드/엣지 스타일, 필터 옵션이 시스템별로 다름

## 8. 성능 고려사항

### 8-1. Explorer (260개 노드)

| 연산 | 복잡도 | 예상 시간 |
|------|--------|----------|
| FAISS pairwise (260×260) | O(n²×d) | ~1초 (1024차원, CPU) |
| 용어집 매칭 (260문서×26K용어) | O(n×m) | ~5초 (문자열 검색) |
| 메뉴 계층 파싱 | O(n) | 즉시 |
| D3 force 시뮬레이션 (260노드) | O(n log n) per tick | 원활 |

- **캐싱으로 해결**: 인덱스 변경 시에만 재계산, 그 외에는 캐시 반환
- Canvas 렌더링 500개 노드까지 60fps 유지 가능

### 8-2. Translator (사용자별 수십 건)

| 연산 | 복잡도 | 예상 시간 |
|------|--------|----------|
| 키워드 추출 (N문서) | O(n×p) (p=페이지) | ~1초 |
| 키워드 공출현 (N×N) | O(n²) | 즉시 (N<100) |
| D3 force (수십 노드) | O(n log n) | 즉시 |

- 문서 수가 적어 성능 문제 없음
- 임베딩 유사도(Phase 6)는 Ollama 호출 필요 → 업로드 시 비동기 처리
